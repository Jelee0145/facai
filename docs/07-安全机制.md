## 1. 认证体系

项目有两套独立的 JWT 认证流程，使用同一个 `JWT_SECRET` 签发。

### 1.1 管理员认证 (Admin)

| 配置项 | 值 |
|--------|-----|
| Token 算法 | HS256 |
| Token 有效期 | 7 天 (10080 分钟) |
| 刷新有效期 | 7 天 |
| 密码哈希 | bcrypt (12 轮) |
| Cookie 名 | `access_token` |
| 存储 | `admin_users` 表 |

**JWT Payload 结构：**
```json
{
  "jti": "uuid — 用于 CSRF Token 和吊销",
  "sub": "admin",
  "role": "admin",
  "iat": 1717200000,
  "exp": 1717286400
}
```

**认证流程：**
```
POST /admin/login
  → verify_password(bcrypt)
  → record_login_attempt(success=True)
  → create_token(username, role)
  → Set-Cookie: access_token=<jwt>; HttpOnly; SameSite=Lax; Path=/
  → 响应体包含 csrf_token (= jti)
```

### 1.2 前台用户认证 (Customer)

| 配置项 | 值 |
|--------|-----|
| Token 算法 | HS256 |
| Token 有效期 | 7 天 (10080 分钟) |
| 密码策略 | 12-128 位，必须含大小写字母+数字+特殊字符 |
| Cookie 名 | `user_access_token` |
| 存储 | `users` 表 |

**JWT Payload 结构：**
```json
{
  "jti": "uuid",
  "sub": "1",
  "username": "testuser",
  "role": "user",
  "token_type": "customer",
  "iat": 1717200000,
  "exp": 1717286400
}
```

**Token 验证逻辑（`verify_token`）：**
1. `jwt.decode()` 校验签名和过期时间
2. 检查 `jti` 是否存在（无 `jti` → 拒绝，fail-closed）
3. 查询 `revoked_tokens` 表，已吊销 → 拒绝
4. 数据库查询失败 → 拒绝（fail-closed）

### 1.3 认证函数对照

| 函数 | 用途 | Cookie | Header |
|------|------|--------|--------|
| `authenticate()` | 管理员端点 | `access_token` | `Authorization: Bearer` |
| `authenticate_customer()` | 用户端点 | `user_access_token` | `Authorization: Bearer` |

两者均支持 Cookie 优先、Header 兜底的方式。

## 2. CSRF 防护

### 设计原理

使用 JWT 的 `jti`（JWT ID）声明作为 CSRF Token。服务端无需额外存储，只需比对请求头中的 Token 与 JWT Payload 中的 `jti` 是否一致。

### 工作流程

```
登录 → 服务端签发 JWT (含 jti)
     → 响应体返回 csrf_token (= jti)
     → 前端保存到内存（state/context）

写请求 → 前端在请求头中添加 X-CSRF-Token: <jti>
       → 后端从 Cookie 解析 JWT → 提取 jti
       → 比对 X-CSRF-Token == jti
       → 不匹配 → 403 CSRF token mismatch
```

### CSRF 校验范围

| 身份 | 校验函数 | 适用方法 |
|------|---------|---------|
| 管理员 | `verify_csrf` (FastAPI Depends) | POST/PUT/DELETE |
| 前台用户 | `_verify_user_csrf` (函数内调用) | POST/PUT/DELETE |

### 安全性分析

- **双重提交 Cookie 模式变体**：CSRF Token 不在 Cookie 中，而在响应体中返回，需要 JavaScript 主动读取并注入请求头
- **HttpOnly Cookie** 中的 JWT 无法被 JavaScript 读取，攻击者无法获取 `jti`
- 即使 Cookie 被自动携带，没有正确的 `X-CSRF-Token` 请求头也会被拒绝

## 3. Cookie 安全属性

### 管理员 Cookie (`access_token`)

```
Set-Cookie: access_token=<jwt>;
  HttpOnly;        ← JavaScript 不可读
  SameSite=Lax;    ← 跨站请求不携带
  Path=/;          ← 全站有效
  Max-Age=604800;   ← 7 天过期
  Secure=<COOKIE_SECURE>;  ← 生产环境启用 HTTPS-only
```

### 用户 Cookie (`user_access_token`)

```
Set-Cookie: user_access_token=<jwt>;
  HttpOnly;
  SameSite=Lax;
  Path=/;
  Max-Age=604800;
  Secure=<COOKIE_SECURE>;
```

### `COOKIE_SECURE` 配置

通过环境变量 `COOKIE_SECURE` 控制（默认 `false`）。生产环境应设为 `true`。

## 4. Token 吊销

### 实现

- 吊销表：`revoked_tokens`（`jti` PRIMARY KEY, `expires_at`）
- 签发时：每个 JWT 包含唯一 `jti`
- 登出时：将当前 JWT 的 `jti` 写入吊销表
- 刷新时：旧 JWT 的 `jti` 被吊销，新 JWT 获得新的 `jti`
- 验证时：每次 `verify_token()` 查询吊销表

### 清理

过期的吊销记录由 `cleanup_expired_revoked_tokens()` 清理（`DELETE FROM revoked_tokens WHERE expires_at < datetime('now')`）。

### 吊销触发点

| 场景 | 行为 |
|------|------|
| `POST /admin/logout` | 吊销当前管理员 Token |
| `POST /admin/refresh` | 吊销旧 Token，签发新 Token |
| `POST /auth/logout` | 吊销当前用户 Token |

## 5. 防暴力破解

### 管理员

- 存储：`admin_users.login_attempts` + `admin_users.locked_until`
- 阈值：连续失败 ≥5 次
- 锁定时间：15 分钟
- 成功登录：重置 `login_attempts=0`，清除 `locked_until`

### 前台用户

- 存储：`users.login_attempts` + `users.locked_until` + `customer_login_attempts` 表
- **双表记录**：对不存在的用户名也通过 `customer_login_attempts` 记录尝试，防止用户名枚举
- 阈值和锁定时间与管理员相同

### 锁定检测流程

```
POST /auth/login
  → get_customer_login_lock(username)  ← 检查 customer_login_attempts 和 users 两表
  → 锁定中 → 429 "账号已被锁定，请 X 分钟后重试"
  → 未锁定 → 验证密码
    → 失败 → record_customer_login_attempt(success=False)
      → 登录尝试 ≥5 → 设置 locked_until = now + 15min
    → 成功 → record_customer_login_attempt(success=True)
```

## 6. 速率限制

### 实现

使用 `slowapi` 库（基于 Redis 内存计数器），按身份标识进行限流。

### 限流 Key 选择策略（`rate_limit_key`）

```
1. 解码 admin Cookie → role=admin → key = "admin:{username}"
2. 解码 user Cookie  → token_type=customer → key = "user:{user_id}"
3. 兜底 → key = 客户端 IP 地址
```

已认证用户按身份限流（避免同 IP 多用户互相影响），未认证请求按 IP 限流。

### 限流规则

| 接口 | 限制 | 说明 |
|------|------|------|
| `POST /auth/register` | 10/min | 防注册轰炸 |
| `POST /auth/login` | 10/min | 配合账号锁定 |
| `POST /admin/login` | 5/min | 后台更严格 |
| `POST /api/generate` | 10/min | 同步生成 |
| `POST /api/generate/async` | 20/min | 异步生成 |
| `GET /api/generate/status/*` | 30/min | 进度查询 |
| 管理端点（写操作） | 10-20/min | 按端点区分 |

## 7. CORS 策略

```python
CORS_ORIGINS = os.getenv("CORS_ORIGINS", "http://localhost:4524")

CORSMiddleware(
    allow_origins=cors_origins,    # 允许的前端域名
    allow_credentials=True,         # 允许携带 Cookie
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Content-Type", "Authorization", "Cookie"],
)
```

**注意：** `allow_origins=["*"]` 与 `allow_credentials=True` 冲突，浏览器会拒绝携带 Cookie 的跨域请求。生产环境必须配置具体的前端域名。

## 8. 输入校验

### 密码策略

| 规则 | 要求 |
|------|------|
| 长度 | 12-128 字符 |
| 小写字母 | 必须包含 |
| 大写字母 | 必须包含 |
| 数字 | 必须包含 |
| 特殊字符 | 必须包含 |

### 图片校验

| 检查项 | 规则 |
|--------|------|
| 格式 | HTTP URL 或 `data:image/*;base64` |
| MIME 类型 | JPEG / PNG / WebP / AVIF |
| 大小 | ≤ 5MB（base64 解码后） |
| Magic Bytes | 校验文件头与声明 MIME 一致 |

### 输入消毒

`sanitize_input(value, max_length)` — strip + 截断，应用于所有用户输入字段。

### Pydantic 校验

所有请求体使用 Pydantic v2 模型，包含：
- `Field(min_length, max_length)` — 字符串长度
- `Field(pattern=...)` — 正则校验（如国家代码、生成类型）
- `Field(ge, le, gt)` — 数值范围

## 9. 内部 API 认证

### X-API-Auth

前端代理层在转发请求到后端时自动注入 `X-API-Auth` 请求头，值为环境变量 `API_AUTH_TOKEN`。

- 生成类端点（`/api/generate/*`）需要此头部
- 后端使用 `hmac.compare_digest()` 进行时间安全比较
- 生产环境缺少 `API_AUTH_TOKEN` 时拒绝启动

### 前后端通信安全

```
浏览器 ──(Cookie)──> Next.js ──(X-API-Auth + Cookie)──> FastAPI
         ↑                    ↑
    用户身份凭证           内部服务凭证
```

## 10. 安全配置检查清单

### 启动时校验

| 检查项 | 严重级别 | 行为 |
|--------|---------|------|
| `JWT_SECRET` 未设置 | CRITICAL | 拒绝启动 (sys.exit) |
| `NODE_ENV=production` + `API_AUTH_TOKEN` 未设置 | CRITICAL | 拒绝启动 |
| `NODE_ENV=production` + 弱密码 + 管理员不存在 | CRITICAL | 拒绝启动 |
| `CORS_ORIGINS=["*"]` + credentials | WARNING | 打印警告，继续运行 |

### 环境变量安全清单

| 变量 | 要求 |
|------|------|
| `JWT_SECRET` | ≥32 字符随机字符串，不在版本控制中 |
| `API_AUTH_TOKEN` | ≥16 字符随机字符串 |
| `ADMIN_PASSWORD` | 满足密码策略，生产环境禁止弱密码 |
| `COOKIE_SECURE` | 生产环境设为 `true` |
| `CORS_ORIGINS` | 具体前端域名，不用 `*` |
