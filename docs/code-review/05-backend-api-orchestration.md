# 模块审查提示词：后端 API 编排

## 模块概述

FastAPI 后端主模块（`main.py`）是整个系统的编排中心，负责图片生成流程、用户认证、管理后台、计费系统等所有 API 端点的实现和协调。

## 涉及文件

| 文件 | 职责 | 行数 |
|------|------|------|
| `backend/main.py` | FastAPI 应用、所有 API 端点、业务编排 | ~1600 |

## 审查提示词

请对后端 API 编排模块进行全面代码审查，重点检查以下方面：

### 1. 架构设计（High）

- **单文件问题**：
  - `main.py` 约 1600 行，是否应该拆分为多个路由模块（router）
  - 建议拆分方向：生成路由、认证路由、管理路由、用户路由、计费路由
  - 是否应该使用 FastAPI 的 `APIRouter` 功能

- **职责分离**：
  - 业务逻辑是否与路由处理混在一起
  - 是否应该抽取 Service 层
  - 图片生成的完整流程（上传 → 匹配 → 构建 prompt → 批量生成 → 轮询）是否应该封装

### 2. 并发安全（Critical）

- **后台任务管理**：
  - Zombie task reaper（60s 间隔）的并发控制
  - 任务清理的原子性
  - 服务重启后进行中的任务如何处理

- **竞态条件**：
  - 余额扣减与退款的原子性（`charge_generation` + `refund_generation`）
  - API Key 使用计数的并发更新
  - 多个请求同时操作同一资源

- **asyncio 安全**：
  - 同步数据库操作是否阻塞事件循环
  - 是否需要使用 `asyncio.to_thread` 包装同步调用
  - `asyncio.create_task` 创建的任务是否正确管理生命周期

### 3. 输入验证（High）

- **图片数据验证**：
  - `validate_image_data` 是否覆盖所有攻击向量
  - data URL 的 MIME 类型验证是否可靠
  - base64 解码后的实际内容检查（magic bytes）

- **用户输入消毒**：
  - `sanitize_input` 是否覆盖所有入口点
  - 商品类型、国家代码等参数的白名单验证
  - SQL 注入防护（虽然使用参数化查询，但动态表名/列名需注意）

- **Pydantic 模型**：
  - 请求模型的字段验证是否严格
  - 是否有缺失的验证规则（如 min_length, max_length, pattern）

### 4. 错误处理（High）

- **全局异常处理器**：
  - 是否泄露敏感信息（堆栈跟踪、内部路径）
  - 路径消毒（path sanitization）是否有效
  - 是否正确处理不同类型的异常

- **外部 API 调用**：
  - Apimart API 调用的错误处理是否完善
  - LLM API 调用的超时和重试
  - 网络错误的降级策略

- **业务异常**：
  - 余额不足、Key 不可用等业务异常是否正确返回
  - 异常消息是否用户友好

### 5. 性能（Medium）

- **同步/异步模式**：
  - 图片生成的同步模式是否会阻塞
  - 异步模式的轮询机制是否高效
  - 批量生成（14 张图）的并发控制

- **数据库操作**：
  - 是否存在 N+1 查询
  - 大量历史记录的分页查询效率
  - 是否需要数据库连接池

- **内存管理**：
  - 图片 base64 数据在内存中的驻留时间
  - 批量生成时的内存峰值

### 6. API 设计（Medium）

- **RESTful 规范**：
  - HTTP 方法使用是否正确
  - 状态码是否准确
  - 响应格式是否一致

- **版本控制**：
  - API 是否需要版本前缀
  - 向后兼容性策略

## 审查输出格式

请按以下格式输出审查结果：

```
### [严重程度] 问题标题

**文件**: `backend/main.py:行号`
**问题**: 问题描述
**建议**: 改进方案
**影响**: 不修改的潜在风险
```

严重程度分级：
- **Critical**: 并发安全或数据损坏风险
- **High**: 功能缺陷或安全问题
- **Medium**: 代码质量或性能问题
- **Low**: 最佳实践建议
