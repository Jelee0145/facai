# 模块审查提示词：后端数据层

## 模块概述

数据层基于 SQLite 实现，提供线程安全的数据库连接、完整的 Schema 定义、CRUD 操作和数据迁移功能。使用 WAL 模式支持并发读写。

## 涉及文件

| 文件 | 职责 | 行数 |
|------|------|------|
| `backend/database.py` | SQLite 连接管理、Schema 初始化、所有 CRUD 操作 | ~940 |

## 审查提示词

请对后端数据层进行全面代码审查，重点检查以下方面：

### 1. 线程安全（Critical）

- **连接管理**：
  - `threading.local()` 在 asyncio 环境下是否安全
  - 是否需要使用 `asyncio.local()` 或 `asyncio.Queue` 管理连接
  - 异步框架中的同步数据库调用是否阻塞事件循环

- **WAL 模式**：
  - WAL 模式配置是否正确（`PRAGMA journal_mode=WAL`）
  - WAL 检查点（checkpoint）是否需要手动管理
  - 并发写入时的性能表现

- **事务管理**：
  - 是否所有写操作都使用事务
  - 嵌套事务的处理
  - 事务超时和回滚

### 2. 数据完整性（High）

- **外键约束**：
  - `PRAGMA foreign_keys=ON` 是否在每个连接上执行
  - 外键约束是否覆盖所有关联关系
  - 级联删除/更新的处理

- **Schema 迁移**：
  - 迁移逻辑是否健壮（多版本跳跃）
  - 是否需要版本号管理
  - 迁移失败的回滚策略

- **数据验证**：
  - Pydantic 模型是否与 Schema 一致
  - 是否需要数据库层面的 CHECK 约束
  - 默认值的一致性

### 3. SQL 注入（High）

- **参数化查询**：
  - 是否所有查询都使用参数化
  - 动态表名/列名是否安全（字符串拼接）
  - LIKE 查询的通转义

- **动态 SQL**：
  - `executescript` 的使用是否安全
  - 动态构建的 SQL 语句
  - 用户输入是否直接拼接到 SQL 中

### 4. 性能（Medium）

- **索引**：
  - 高频查询的 WHERE 条件是否有索引
  - `task_id`、`user_id`、`order_no` 等字段的索引
  - 复合索引的使用

- **查询优化**：
  - 是否存在 N+1 查询
  - 大量历史记录的分页查询效率
  - COUNT 查询的性能

- **连接池**：
  - 当前使用线程本地连接，是否需要连接池
  - 长时间运行的查询是否会占用连接

### 5. 数据安全（Medium）

- **敏感数据存储**：
  - API Key `key_value` 是否加密存储（当前明文）
  - 密码哈希的存储安全
  - 个人信息（手机号、邮箱）的保护

- **数据备份**：
  - 是否有自动备份机制
  - 备份文件的安全存储
  - 恢复测试

- **数据清理**：
  - 旧数据的归档/删除策略
  - 敏感数据的定期清理
  - 软删除 vs 硬删除

### 6. 代码质量（Medium）

- **文件大小**：
  - 940 行单文件是否需要拆分为多个 Repository 模块
  - 建议拆分：按功能域（用户、订单、Key、历史）

- **错误处理**：
  - CRUD 函数的错误处理是否一致
  - 异常类型是否明确
  - 是否需要自定义异常类

- **类型安全**：
  - SQLite Row 的类型访问
  - Optional 字段的处理
  - 返回值的类型标注

### 7. 可维护性（Low）

- **命名规范**：
  - 表名和列名的命名约定
  - 函数命名是否一致
  - 常量定义

- **文档**：
  - Schema 的注释
  - 复杂查询的注释
  - 数据字典

## 审查输出格式

请按以下格式输出审查结果：

```
### [严重程度] 问题标题

**文件**: `backend/database.py:行号`
**问题**: 问题描述
**建议**: 改进方案
**影响**: 不修改的潜在风险
```

严重程度分级：
- **Critical**: 数据损坏或并发安全风险
- **High**: 数据完整性或安全问题
- **Medium**: 性能或代码质量问题
- **Low**: 可维护性建议
