# 模块审查提示词：后端 API Key 管理

## 模块概述

API Key 负载均衡管理器支持多 Key 轮询、故障自动禁用和每日限额管理。提供线程安全的 Key 选择和健康检查功能。

## 涉及文件

| 文件 | 职责 | 行数 |
|------|------|------|
| `backend/key_manager.py` | 多 API Key 轮询、故障转移、健康检查 | 66 |

## 审查提示词

请对 API Key 管理模块进行全面代码审查，重点检查以下方面：

### 1. 并发安全（Critical）

- **锁机制**：
  - `threading.Lock()` 在 asyncio 环境下是否正确
  - 是否应该使用 `asyncio.Lock` 或 `asyncio.Semaphore`
  - 锁的粒度是否合理（整个 `get_active_key` 持锁期间）

- **轮询索引**：
  - `self._current_index` 在多线程/多进程下是否安全
  - 索引溢出处理（`len(keys)` 变化时）
  - 是否应该使用 `itertools.cycle` 替代手动索引

- **状态一致性**：
  - `get_active_keys()` 返回结果与实际状态的一致性
  - Key 被禁用后是否立即生效

### 2. 故障转移（High）

- **失败检测**：
  - 连续 3 次失败自动禁用是否合理
  - 是否应该区分失败类型（网络错误 vs 认证错误 vs 限流）
  - 临时故障 vs 永久故障的判断

- **恢复机制**：
  - 被禁用的 Key 如何重新启用
  - 是否需要手动干预
  - 自动恢复策略（如每日重置）

- **健康检查**：
  - `health_check()` 是否应该主动探测 Key 可用性
  - 检查频率和超时设置
  - 检查结果的缓存

### 3. 负载均衡（Medium）

- **轮询策略**：
  - 简单轮询是否考虑了各 Key 的限额和性能差异
  - 是否需要加权轮询
  - 是否需要最少连接策略

- **限额管理**：
  - 轮询时是否跳过接近限额的 Key
  - 限额重置的时间点（UTC vs 本地时间）
  - 限额用完后的处理

- **失败计数**：
  - `fail_count` 是否需要衰减机制
  - 成功是否应该减少 fail_count
  - 阈值是否应该可配置

### 4. 可观测性（Medium）

- **监控指标**：
  - 各 Key 的使用率
  - 失败率和成功率
  - 平均响应时间

- **告警**：
  - Key 被禁用时是否需要告警
  - 所有 Key 都不可用时的处理
  - 限额接近时的预警

- **日志**：
  - Key 选择的日志
  - 失败事件的日志
  - 状态变化的日志

### 5. 代码质量（Low）

- **单例模式**：
  - `key_manager` 全局单例是否线程安全
  - 是否需要依赖注入替代全局变量

- **接口设计**：
  - `get_active_key` 返回 dict 是否应该有明确的类型定义
  - 是否应该返回 Key 对象而非原始字典

- **测试**：
  - 是否有单元测试覆盖
  - 并发场景的测试

## 审查输出格式

请按以下格式输出审查结果：

```
### [严重程度] 问题标题

**文件**: `backend/key_manager.py:行号`
**问题**: 问题描述
**建议**: 改进方案
**影响**: 不修改的潜在风险
```

严重程度分级：
- **Critical**: 并发安全或数据损坏风险
- **High**: 故障转移缺陷
- **Medium**: 负载均衡或可观测性问题
- **Low**: 代码质量建议
