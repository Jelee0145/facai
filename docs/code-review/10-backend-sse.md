# 模块审查提示词：后端实时通信

## 模块概述

SSE（Server-Sent Events）模块提供实时进度推送能力，基于 asyncio.Queue 实现发布-订阅模式。支持多客户端订阅同一任务的进度事件。

## 涉及文件

| 文件 | 职责 | 行数 |
|------|------|------|
| `backend/sse.py` | SSE Pub/Sub、asyncio.Queue、超时管理 | 50 |

## 审查提示词

请对后端实时通信模块进行全面代码审查，重点检查以下方面：

### 1. 内存管理（High）

- **订阅者清理**：
  - 客户端断开后订阅者是否正确清理
  - `unsubscribe` 的调用时机是否可靠
  - `finally` 块是否保证执行

- **内存泄漏**：
  - 长期未清理的 `task_id` 是否会累积
  - 是否需要定期清理过期的订阅者
  - `_subscribers` 字典的大小是否有限制

- **Queue 管理**：
  - `QueueFull` 时静默丢弃事件是否合适
  - 是否需要丢弃旧事件而非新事件
  - 队列大小是否应该有限制

### 2. 并发安全（High）

- **数据结构**：
  - `defaultdict(list)` 在异步环境下是否线程安全
  - `subscribe`/`unsubscribe` 的并发调用是否安全
  - 是否需要锁保护

- **迭代安全**：
  - `push_event` 遍历订阅者列表时，其他协程是否可能修改列表
  - 是否需要复制列表后遍历

- **asyncio 兼容性**：
  - 当前实现是否与 FastAPI 的 asyncio 环境兼容
  - 是否需要使用 `asyncio.Queue` 替代普通 Queue

### 3. 超时处理（Medium）

- **超时配置**：
  - 300 秒超时是否合理（生成任务可能更长）
  - 超时时间是否应该可配置
  - 超时后的用户引导

- **资源清理**：
  - 超时后 Queue 是否正确清理
  - 订阅者是否正确移除
  - 是否需要通知生产者停止生成

### 4. 可靠性（Medium）

- **服务重启**：
  - 进行中的任务如何恢复
  - 是否需要持久化事件队列
  - 任务状态的恢复机制

- **消息顺序**：
  - 事件是否保证按顺序送达
  - 是否需要序列号
  - 乱序处理

- **错误处理**：
  - `push_event` 中的异常处理
  - 生成器中的异常处理
  - 客户端异常的处理

### 5. 可观测性（Low）

- **监控指标**：
  - 活跃订阅者数量
  - 事件推送延迟
  - 队列深度

- **日志**：
  - 订阅/取消订阅事件
  - 超时事件
  - 错误事件

### 6. 代码质量（Low）

- **接口设计**：
  - `push_event` 是同步函数，在异步环境中是否合适
  - 是否应该使用 `asyncio.Queue.put_nowait` 而非 `q.put_nowait`

- **类型安全**：
  - `data: dict` 的类型标注
  - Queue 的类型参数

- **测试**：
  - 是否有单元测试
  - 并发场景的测试

## 审查输出格式

请按以下格式输出审查结果：

```
### [严重程度] 问题标题

**文件**: `backend/sse.py:行号`
**问题**: 问题描述
**建议**: 改进方案
**影响**: 不修改的潜在风险
```

严重程度分级：
- **Critical**: 内存泄漏或并发安全风险
- **High**: 可靠性或功能缺陷
- **Medium**: 代码质量或性能问题
- **Low**: 最佳实践建议
