## 2. 双服务架构

| 层 | 技术 | 端口 | 职责 |
|----|------|------|------|
| 前端 | Next.js 16 + React 19 | 4524 | UI 渲染、API 代理、Cookie 注入 |
| 后端 | FastAPI + SQLite | 8001 | 业务逻辑、图片生成、用户管理 |
| 图片生成 | Apimart AI API | 外部 | gpt-image-2 模型，14 张图/任务 |
| LLM 分析 | 阿里云 DashScope | 外部 | Qwen3 VL 智能 prompt/品类分析 |

### 为什么不直接让前端调后端？

前端通过 Next.js API Route 代理后端请求，原因：
1. **安全隔离**：`X-API-Auth` 内部令牌不暴露给浏览器
2. **CSRF 防护**：代理层统一注入 CSRF Token
3. **熔断保护**：代理层集成 CircuitBreaker，防止后端故障蔓延
4. **Cookie 管理**：代理层处理 Cookie 转发和 Set-Cookie 回传

## 4. 模块依赖关系

```
main.py (FastAPI 入口，全部端点定义)
├── database.py      ← 数据库 CRUD (SQLite, 线程安全)
├── security.py      ← JWT 签发/验证 + bcrypt + 防暴力破解
├── csrf.py          ← CSRF Token 校验
├── middleware.py     ← slowapi 速率限制
├── sse.py           ← SSE 异步队列 pub/sub
├── key_manager.py   ← 多 API Key 轮询 + 故障转移
├── llm_provider.py  ← DashScope LLM 调用 + 降级
└── prompts_v2.py    ← 品类匹配 + 模型选择 + prompt 构建
```

### 关键模块说明

#### key_manager.py — API Key 轮询

- **线程安全**：`threading.Lock` 保护轮询索引
- **轮询策略**：Round-Robin，按 `last_used_at ASC` 排序
- **故障转移**：401/403 响应 → `mark_failure()` → 连续失败 ≥3 次自动禁用
- **配额管理**：每日 `today_used` 自增，超 `daily_limit` 跳过
- **健康检查**：`health_check()` 返回每个 Key 的使用率和余额

#### llm_provider.py — LLM 分析

- 调用阿里云 DashScope API（默认 Qwen3 VL Flash）
- **降级策略**：LLM 调用失败时静默降级为模板 prompt，不影响主流程
- 配置通过 `system_config` 表的 `llm_api_key` / `llm_model` 管理

#### sse.py — 实时推送

- 基于 `asyncio.Queue` 的 pub/sub 模式
- `push_event(task_id, data)` → 向所有订阅者广播
- `sse_stream(task_id)` → 生成 SSE 数据帧，空闲超时 300 秒
- 终态（completed/failed/error）事件保证送达：队列满时丢弃最旧非终态事件

#### prompts_v2.py — Prompt 构建

- 品类匹配：根据用户输入的产品类型匹配最佳品类模板
- 模型选择：根据品类和用户选择确定模型风格
- Prompt 生成：结合品类、国家、模型、LLM 分析结果构建 14 个子任务

### 5.2 API 代理层

所有 `/api/*` 路由统一使用 `proxyToBackend()` 函数（`src/lib/proxy.ts`）：

| 配置项 | 说明 |
|--------|------|
| `targetPrefix` | 替换路径前缀，如 `/api/auth` → 后端 `/auth` |
| `forwardCookies` | 是否转发 Cookie（登录路由需要） |
| `forwardSetCookie` | 是否回传 Set-Cookie（登录响应需要） |
| `handleStreaming` | 是否支持 SSE 流式转发 |

### 5.3 网络韧性

```
fetchWithRetry          CircuitBreaker
(指数退避重试)           (熔断保护)
  │                       │
  │ 502/503/504 重试 3 次  │ 连续失败 ≥5 次 → OPEN
  │ 延迟: 1s → 2s → 4s    │ 30 秒后 → HALF_OPEN
  │                       │ 成功 → CLOSED
  └───────┬───────────────┘
          │
          ▼
     proxyToBackend()
```

### 5.4 SSE Hook

`useSSE(taskId, handlers)` 封装了 EventSource 的完整生命周期：

- 自动连接：taskId 非空时自动创建 EventSource
- 自动重连：断开后重试 3 次，间隔 5 秒
- 会话检测：重连前检查 `/auth/me`，过期则停止重连
- 自动清理：组件卸载时关闭连接

## 6. 数据存储

### SQLite 配置

- **WAL 模式**：支持并发读写，读操作不阻塞写操作
- **线程本地连接**：每个线程独立连接，通过 `threading.local()` 管理
- **外键约束**：`PRAGMA foreign_keys=ON`
- **迁移策略**：应用内增量迁移（`ALTER TABLE ADD COLUMN`）

### 数据分层

| 层 | 表 | 特点 |
|----|-----|------|
| 用户系统 | `users`, `user_wallets`, `orders`, `user_ledger`, `payment_transactions` | 用户注册/充值/消费 |
| 管理系统 | `admin_users`, `api_keys`, `system_config`, `credit_packages` | 后台管理 |
| 生成系统 | `generation_history`, `task_store`, `custom_product_types` | 图片生成 |
| 安全系统 | `revoked_tokens`, `customer_login_attempts` | 认证安全 |

## 8. 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 前后端分离 | Next.js + FastAPI | 前端需要 SSR 能力，后端需要 Python 生态（AI/ML 库） |
| 数据库 | SQLite + WAL | 轻量部署，单机场景足够，WAL 支持并发读写 |
| 异步生成 | 后台协程 + SSE | 14 张图生成耗时长，必须异步 + 实时推送 |
| 多 Key 轮询 | Round-Robin + 自动禁用 | Apimart 有配额限制，多 Key 分散压力 |
| LLM 分析 | 可选 + 降级 | LLM 不影响核心生成流程，失败时静默降级 |
| 前端代理 | Next.js API Route | 隐藏内部令牌，统一 CSRF/限流/Cookie 管理 |
| 熔断器 | 前端 CircuitBreaker | 后端故障时快速失败，避免前端雪崩 |
